## 背景：8,000人組織での課題

### 環境

| 項目 | 値 |
| --- | --- |
| 連結従業員数 | 8,000人以上 |
| 連結子会社数 | 86社（2025年9月末時点） |
| Slackワークスペース | 複数（子会社ごとに分離） |
| 技術選定 | 各チーム独自（ボトムアップ文化） |

### 問題

2023年末からCursorを使い始め、2024年9月に社内Slackチャンネルを作成。しかし：

- 社内登壇しても反応が薄い（「へー、便利そうですね」で終わり）
- ドキュメント整備しても読まれない
- 部署の壁を越えて情報が届かない

**結論**: 良いツールを紹介するだけでは、大企業では浸透しない。

---

## 戦略：外部から内部を攻める

### なぜ外部イベントか

```
内部アプローチの限界:
  ├── Slackワークスペースの壁
  ├── 部署・子会社の境界
  ├── 「またツール推進か」という認知バイアス
  └── 意思決定者へのリーチが困難

外部アプローチの利点:
  ├── Xトレンド、YouTube話題は全員に同時に届く
  ├── 外部の評価 = 社会的証明として機能
  ├── 社内の壁を飛び越える
  └── アウトプット品質が自動的に上がる（外部の目があるため）
```

## 7つの実践テクニック

### ① 影響力のある人をDMで巻き込む

人脈ゼロからスタート。やったことはシンプルに **DMを送る** 。

**効果的なDMの構成:**

```markdown
## DMテンプレート

〇〇さん

突然のDM失礼いたします。サイバーエージェントの△△です。

〇〇さんの「{具体的な発信内容を引用}」を拝見し、
特に「{共感ポイント}」という視点に深く共感しました。

現在、AIコードエージェントをテーマにした技術イベントを企画しています。
- 想定参加者: {数値}名（現地+オンライン）
- 形式: {登壇時間}分のセッション or LT
- 日程: {候補日}

ご検討いただけますと幸いです。
```

**ポイント:**

- テンプレ感を消す（相手の発信を具体的に引用）
- 価値を数値で示す（参加者規模、アーカイブ視聴数など）
- 依頼を小さくする（時間、形式、負担を明確に）

---

### ② ペルソナを設計して波及経路を作る

「エンジニア向けイベント」で終わらせない。波及を起こす設計を最初から埋め込む。

| カテゴリ | 波及の役割 | 具体例 |
| --- | --- | --- |
| 経営層 | 意思決定への影響力 | CTO、技術担当役員 |
| 技術インフルエンサー | 外部からの集客・社会的証明 | フォロワー数が多い実践者 |
| 社外実践者 | 再現性の核（「他社でもやってる」） | 他社のAI導入担当者 |
| 海外CEO | 一次情報 + 話題性 | Cursor社、関連スタートアップ |
| 社内実践者 | 「うちでもできる」の証拠 | 各事業部のアーリーアダプター |
| 非エンジニア | 民主化の体現 | デザイナー、PM |

---

### ③ LP・ビジュアルの技術スタック

**使用ツール:**

| 用途 | ツール | 選定理由 |
| --- | --- | --- |
| LP制作 | Canva | 非エンジニアでも高速に制作可能 |
| ビジュアル生成 | AI画像生成（Midjourney等） | オリジナル素材を大量生成 |
| ポスターレイアウト | Adobe Illustrator | 印刷物のベクター処理 |
| 画像アップスケール | Topaz Labs | 3mポスター用の高解像度化 |

**なぜデザインに投資するか:**

- LPは「情報の置き場」ではなく「信頼と熱量を伝えるメディア」
- 外部の人が見た瞬間に「ちゃんとしてる」と感じる必要がある

---

### ④ 言語の壁をAIで解決

海外登壇者を招くために、 **Cuckoo** （AIリアルタイム翻訳）を導入。

**機能:**

- 英語↔日本語の同時通訳
- 話者が言語を切り替えても追従
- 参加者はスマホで翻訳音声を受信

アンケートで「リアルタイムAI翻訳がすごかった」という声が多数。

### ⑤ オンライン配信の技術構成

### ⑦ マルチエージェント戦略

一つのツールに固定しない。複数のAIコードエージェントを使い分け。

**現在使用中のツール:**

| ツール | 主な用途 |
| --- | --- |
| Cursor | 日常の開発、エディタ統合が必要な作業 |
| Claude Code | 大規模なリファクタリング、複雑な推論 |
| Codex | コード生成、特定タスクの自動化 |

**なぜマルチベンダーか:**

- ツールの進化速度が速い（月1回は再評価）
- タスクによって最適なツールが異なる
- ベンダーロックインを避ける

## 再現プレイブック

### Week 1: 基盤構築

```bash
Day 1: Slackチャンネル作成
Day 2-3: 自分が「効いた」Tipsを投稿
Day 4-5: 興味持ちそうな同僚に1on1で声かけ
Day 6-7: 15分のミニ勉強会を企画
```

### Month 1-3: コンテンツ蓄積

- 週1でTipsを投稿
- 海外の事例をキュレーション
- 質問があれば即レス（初期の反応速度が重要）

### Month 3-6: イベント企画

- 登壇者候補にDM（10人に送って2-3人から返信があれば上出来）
- LP制作（Canvaで十分）
- connpassで募集開始

## うまくいかなかったこと

成功談だけでは再現性が落ちるので、失敗も共有。

| 失敗 | 学び |
| --- | --- |
| 最初の社内登壇は響かなかった | 技術説明 < ストーリー（なぜ今・なぜ自分たち） |
| ドキュメント整備だけでは広まらない | 読む人は限られる。体験の導線が必要 |
| 全員に刺さるメッセージはない | 層ごとに価値が違う（下記参照） |

**層別の訴求ポイント:**

| ターゲット | 刺さるメッセージ |
| --- | --- |
| 経営層 | 生産性向上、コスト削減 |
| ベテランエンジニア | 開発体験の向上 |
| 若手エンジニア | キャリアの武器になる |
| 非エンジニア | プロトタイピングが可能 |
